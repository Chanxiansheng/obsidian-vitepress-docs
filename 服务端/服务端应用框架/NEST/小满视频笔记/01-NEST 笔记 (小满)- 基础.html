简介 ​

Nest 是一个用于构建高效，可扩展的 Node.js 服务器端应用程序的框架。它使用渐进式 JavaScript，内置并完全支持 TypeScript（但仍然允许开发人员使用纯 JavaScript 编写代码）并结合了 OOP（面向对象编程），FP（函数式编程）和 FRP（函数式响应编程）的元素。

在底层，Nest 使用强大的 HTTP Server 框架，如 ==Express==（默认）和 ==Fastify==。Nest 在这些框架之上提供了一定程度的抽象，同时也将其 API 直接暴露给开发人员。这样可以轻松使用每个平台的无数第三方模块。

具体见 express基础

前置知识 ​

控制反转与依赖注入 ​

控制反转（Inversion of Control，IoC）和依赖注入（Dependency Injection，DI）是软件开发中常用的设计模式和技术，用于解耦和管理组件之间的依赖关系。虽然它们经常一起使用，但它们是不同的概念。

  • 控制反转（IoC）是一种设计原则，它将组件的控制权从组件自身转移到外部容器。传统上，组件负责自己的创建和管理，而控制反转则将这个责任转给了一个外部的容器或框架。容器负责创建组件实例并管理它们的生命周期，组件只需声明自己所需的依赖关系，并通过容器获取这些依赖。这种反转的控制权使得组件更加松耦合、可测试和可维护。

  • 依赖注入（DI）是实现控制反转的一种具体技术。它通过将组件的依赖关系从组件内部移动到外部容器来实现松耦合。==组件不再负责创建或管理它所依赖的其他组件，而是通过构造函数、属性或方法参数等方式将依赖关系注入到组件中。==依赖注入可以通过构造函数注入（Constructor Injection）、属性注入（Property Injection）或方法注入（Method Injection）等方式实现。

这是依赖注入的实现案例：

ts
// 引入元数据支持
import 'reflect-metadata';

// ================= 1. 定义抽象接口 =================
interface IA {
  name: string;
}

interface IC {
  name: string;
}

// ================= 2. 实现具体类 =================
class ServiceA implements IA {
  constructor(public name: string) {}
}

class ServiceC implements IC {
  constructor(public name: string) {}
}

// ================= 3. 依赖注入容器 =================
const TYPES = {
  IA: Symbol('IA'),
  IC: Symbol('IC'),
  IB: Symbol('IB')
};

class Container {
  private providers = new Map<symbol, any>();

  // 注册依赖关系
  register<T>(identifier: symbol, provider: new (...args: any[]) => T) {
    this.providers.set(identifier, provider);
  }

  // 递归解析依赖
  resolve<T>(identifier: symbol): T {
    const Target = this.providers.get(identifier);
    
    // 获取构造函数参数类型
    const paramTypes: symbol[] = Reflect.getMetadata('design:paramtypes', Target) || [];
    
    // 递归解析所有依赖
    const args = paramTypes.map((type: symbol) => this.resolve(type));
    
    return new Target(...args);
  }
}

// ================= 4. 业务类定义 =================
class ServiceB {
  constructor(
    private readonly serviceA: IA,
    private readonly serviceC: IC
  ) {}

  display() {
    console.log(`B依赖的服务：${this.serviceA.name} 和 ${this.serviceC.name}`);
  }
}

// ================= 5. 配置依赖关系 =================
const container = new Container();

// 注册实现类（可在不修改业务类的情况下替换实现）
container.register(TYPES.IA, ServiceA);
container.register(TYPES.IC, ServiceC);
container.register(TYPES.IB, ServiceB);

// ================= 6. 使用容器 =================
// 获取实例（自动注入依赖）
const serviceB = container.resolve<ServiceB>(TYPES.IB);

// 设置初始参数（可通过工厂模式进一步优化）
(serviceB as any).serviceA.name = "A服务实例";
(serviceB as any).serviceC.name = "C服务实例";

serviceB.display(); // 输出：B依赖的服务：A服务实例 和 C服务实例

装饰器 ​

具体参考：D-学习笔记/前端技术/05 TypeScript/TS(第二轮-小满)/TypeScript(进阶) > 装饰器 Decorators

  • 什么是装饰器？
    装饰器就是一个方法，可以注入到类、方法、属性参数上来拓展它们的功能。
    常用的装饰器有：类装饰器、属性装饰器、方法装饰器、参数装饰器
    装饰器的写法有：普通装饰器（无参），装饰器工厂（有参）

  • 如何启用？
    若要启用实验性的装饰器特性，你必须在命令行或 tsconfig.json 里启用 experimentalDecorators 编译器选项

  • 语法
    装饰器是一种特殊类型的声明，它能够被附加到类声明，方法，访问符，属性或参数上。 装饰器使用 @expression 这种形式，expression 求值后必须为一个函数，它会在运行时被调用，被装饰的声明信息做为参数传入。

ts
// 类装饰器
const classdoc: ClassDecorator = (target: any) => {
  console.log("类装饰器", target)
}

// 属性装饰器
const propertydoc: PropertyDecorator = (target: any, key: string | symbol) => {
  console.log("属性装饰器", target, key)
}

// 方法装饰器
const methoddoc: MethodDecorator = (target: any, key: string | symbol, descriptor: PropertyDescriptor) => {
  console.log("方法装饰器", target, key, descriptor)
}

// 参数装饰器
const paramdoc: ParameterDecorator = (target: any, key: string | symbol | undefined, index: number) => {
  console.log("参数装饰器", target, key, index);
  return target
}

@classdoc
class Person {
  @propertydoc
  public name: string
  constructor() {
    this.name = "william"
  }
  @methoddoc
  getName(@paramdoc name: string) {
    return this.name
  }
}

RESTful 风格设计 ​

RESTful 是一种风格，在 RESTful 中，一切都被认为是资源，每个资源有对应的 URL 标识，不是标准也不是协议，只是一种风格。

ts
// 传统接口 通过路由和query参数
http://localhost:8080/api/get_list?id=1
http://localhost:8080/api/delete_list?id=1
http://localhost:8080/api/update_list?id=1

RESTful 风格用一个接口就会完成增删改差，它是通过不同的请求方式来区分的。

  • 查询 GET
  • 提交 POST
  • 更新 PUT PATCH
  • 删除 DELETE

RESTful 版本控制

URI Versioning版本将在请求的 URI 中传递（默认）
Header Versioning自定义请求标头将指定版本
Media Type Versioning请求的 Accept 标头将指定版本
ts
// main.ts
  app.enableVersioning({
    type: VersioningType.URI
  })

然后在 user.controller 配置版本
http://localhost:3000/user/1 => http://localhost:3000/v1/user/1

ts
// user.controller.ts
@Controller({
  path: "user",
  version: '1'
})

Code 码规范 ​

状态码描述详细说明
200OK请求成功，服务器已成功处理了请求，并返回了请求的资源。
304Not Modified 协商缓存了请求的资源未被修改，客户端可以使用之前缓存的版本。
400Bad Request 参数错误服务器无法理解请求，通常是因为请求参数格式不正确或缺少必要的参数。
401Unauthorized token 错误请求未通过身份验证，通常是因为缺少有效的认证信息（如 token）。
403Forbidden referer origin 验证失败服务器理解请求，但拒绝执行，通常是因为权限不足或请求来源不被允许。
404Not Found 接口不存在服务器找不到请求的资源，通常是因为请求的 URL 路径不正确。
500Internal Server Error 服务端错误服务器内部错误，无法完成请求，通常是由于服务器配置或代码问题。
502Bad Gateway 上游接口有问题或者服务器问题服务器作为网关或代理，但无法从上游服务器获取有效响应。

Session ​

session 是服务器为每个用户的浏览器创建的一个会话对象，当有请求进来， session 会发送一个包含 sessionID 的 cookie 给客户端，当客户端需要做登录等操作时，把 sessionID 返回给服务端进行校验，进而进行用户的校验。

nestjs 默认框架 express ，默认支持 express 的插件，所以我们就可以安装 express 的 session。
npm i express-session --save

需要智能提示可以装一个声明依赖
npm i @types/express-session -D

然后在 main.ts 引入 通过 app.use 注册 session

ts
import * as session from 'express-session'
app.use(session({
    secret: "william",
    rolling: true,
    name: "william.sid",
    cookie: {
      maxAge: 36000
    }
  }))
  • 参数配置详解
key说明
secret生成服务端 session 签名（可以理解为加盐）
name生成客户端 cookie 的名字，默认 connect.sid
cookie设置返回到前端 key 的属性，默认值为{ path: ‘/’, httpOnly: true, secure: false, maxAge: null }。
rolling在每次请求时强行设置 cookie，这将重置 cookie 过期时间 (默认:false)

控制器 Controller ​

控制器负责处理传入的请求和向客户端返回响应。
控制器的目的是接收应用的特定请求。路由机制控制哪个控制器接收哪些请求。通常，每个控制器有多个路由，不同的路由可以执行不同的操作。

路由 ​

import { Controller, Get } from '@nestjs/common';

  @Get()
  findAll(): string {
    return 'This action returns all user';
  }
}

提供者 Providers ​

Providers 是 Nest 的一个基本概念。许多基本的 Nest 类都可能被视为 provider - service, repository, factory, helper 等等。 他们都可以通过 constructor 注入依赖关系。 这意味着对象可以彼此创建各种关系，并且“连接”对象实例的功能在很大程度上可以委托给 Nest 运行时系统。 Provider 只是一个用 @Injectable() 装饰器注释的类。

app.module.ts 引入 app.service.ts  在 providers 注入后，在 app.controller.ts 就可以使用注入好的 service。

1. 默认写法 ​
ts
// module.ts
import { Module } from '@nestjs/common';

import { AppService } from './app.service';
@Module({
  controllers: [AppController],
  providers: [AppService],   //⭐默认写法
})
export class AppModule {}


// service.ts
import { Injectable } from '@nestjs/common';
@Injectable()
export class AppService {
}

// controller.ts
@Controller()
export class AppController {
  constructor(private readonly appService: AppService) {}
}

模块 Module ​

模块是具有 @Module() 装饰器的类。 @Module() 装饰器提供了元数据，Nest 用它来组织应用程序结构。
每个 Nest 应用程序至少有一个模块，即根模块。根模块是 Nest 开始安排应用程序树的地方。事实上，根模块可能是应用程序中唯一的模块，特别是当应用程序很小时，但是对于大型程序来说这是没有意义的。在大多数情况下，您将拥有多个模块，每个模块都有一组紧密相关的功能。

@Module() 装饰器接受一个描述模块属性的对象：

key说明
providers由 Nest 注入器实例化的提供者，并且可以至少在整个模块中共享
controllers必须创建的一组控制器
imports导入模块的列表，这些模块导出了此模块中所需提供者
exports由本模块提供并应在其他模块中可用的提供者的子集。

动态模块 ​

Nest 模块系统包括一个称为动态模块的强大功能。此功能使您可以轻松创建可自定义的模块，这些模块可以动态注册和配置提供程序。
动态模块主要就是为了给模块传递参数，可以给该模块添加一个静态方法用来接受参数。（类似一个类工厂）

ts
import { Module, DynamicModule } from '@nestjs/common';
import { createDatabaseProviders } from './database.providers';
import { Connection } from './connection.provider';

@Module({
  providers: [Connection],
})
export class DatabaseModule {
  static forRoot(entities = [], options?): DynamicModule {
    const providers = createDatabaseProviders(options, entities);
    return {
      module: DatabaseModule,
      providers: providers,
      exports: providers,
    };
  }
}

中间件 ​

中间件是在路由处理程序 之前 调用的函数。 中间件函数可以访问请求和响应对象，以及应用程序请求响应周期中的 next() 中间件函数。 next() 中间件函数通常由名为 next 的变量表示。

中间件函数可以执行以下任务:

  1. 执行任何代码。
  2. 对请求和响应对象进行更改。
  3. 结束请求 - 响应周期。
  4. 调用堆栈中的下一个中间件函数。
  5. 如果当前的中间件函数没有结束请求 - 响应周期, 它必须调用 next() 将控制传递给下一个中间件函数。否则, 请求将被挂起。
中间件的声明 ​

您可以在函数中或在具有 @Injectable() 装饰器的类中实现自定义 Nest 中间件。 这个类应该实现 NestMiddleware 接口, 而函数没有任何特殊的要求。 让我们首先使用类方法实现一个简单的中间件功能。

ts
import { Injectable, NestMiddleware } from "@nestjs/common";
import { Request, Response, NextFunction } from 'express';
@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    console.log('中间件被调用了');
    next();
  }
}
中间件的应用 ​

中间件不能在 @Module() 装饰器中列出。我们必须使用模块类的 configure() 方法来设置它们。包含中间件的模块必须实现 NestModule 接口。我们将 LoggerMiddleware 设置在 ApplicationModule 层上。

ts
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import { UserService } from './user.service';
import { UserController } from './user.controller';
import { LoggerMiddleware } from 'src/middleware';

@Module({
  controllers: [UserController],
  providers: [UserService],
  exports: [UserService]
})
export class UserModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer.apply(LoggerMiddleware).forRoutes('user');
  }
}

MiddlewareConsumer 是一个帮助类。它提供了几种内置方法来管理中间件。他们都可以被简单地链接起来。forRoutes() 可接受一个字符串、多个字符串、对象、一个控制器类甚至多个控制器类。在大多数情况下，您可能只会传递一个由逗号分隔的控制器列表。

可以在配置中间件时将包含路由路径的对象和请求方法传递给 forRoutes() 方法，从而进一步将中间件限制为特定的请求方法。路由也同样支持模式匹配。

ts
forRoutes({ path: 'ab*cd', method: RequestMethod.ALL });

Multer 是一个 Node.js 中间件，用于处理 multipart/form-data 类型的表单数据，主要用于上传文件。它是 Express 和其他基于 Node.js 的 web 框架的常用文件上传中间件。

+ RxJs

RxJS 是一个用于处理异步和事件驱动编程的库，它基于可观察序列（Observables）的概念，提供了一套丰富的操作符来组合、转换、过滤和操作这些序列。RxJS 的核心思想是将数据流视为一个整体，通过声明式的方式对数据流进行操作，从而简化异步编程的复杂性。

拦截器 ​

拦截器是使用 @Injectable() 装饰器注解的类。拦截器应该实现 NestInterceptor 接口。

拦截器具有一系列有用的功能，这些功能受面向切面编程（AOP）技术的启发。它们可以：

  • 在函数执行之前/之后绑定额外的逻辑
  • 转换从函数返回的结果
  • 转换从函数抛出的异常
  • 扩展基本函数行为
  • 根据所选条件完全重写函数 (例如, 缓存目的)

每个拦截器都有 intercept() 方法，它接收 2 个参数。

  • 第一个是 ExecutionContext 实例（与守卫完全相同的对象）。 ExecutionContext 继承自 ArgumentsHost 。 ArgumentsHost 是传递给原始处理程序的参数的一个包装 ，它根据应用程序的类型包含不同的参数数组。
  • 第二个参数是 CallHandler。基本上，CallHandler 是一个包装执行流的对象，因此推迟了最终的处理程序执行。只有 handle() 被调用（并且已返回值），最终方法才会被触发。handle() 返回一个 Observable。

响应拦截器 ​

案例：返回一个标准的 json 格式

ts
import { Injectable, NestInterceptor, CallHandler } from '@nestjs/common'
import { map } from 'rxjs/operators'
import {Observable} from 'rxjs'
 
interface data<T>{
    data:T
}
 
@Injectable()
export class Response<T = any> implements NestInterceptor {
    intercept(context, next: CallHandler):Observable<data<T>> {
        return next.handle().pipe(map(data => {
            return {
               data,
               status:0,
               success:true,
               message:"success"
            }
        }))
    }
}

在 main.ts 全局注册

ts
app.useGlobalInterceptors(new Response())

异常过滤器 ​

异常过滤器可以控制精确的控制流以及将响应的内容发送回客户端。
为此，我们需要访问底层平台 Request 和 Response。我们将访问 Request 对象，以便提取原始 url 并将其包含在日志信息中。我们将使用 Response.json() 方法，使用 Response 对象直接控制发送的响应。

ts
import { ArgumentsHost, Catch, ExceptionFilter, HttpException } from "@nestjs/common";
import { Request, Response } from "express";

@Catch()
export class HttpFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const ctx = host.switchToHttp()
    const request = ctx.getRequest<Request>()
    const response = ctx.getResponse<Response>()
    const status = exception.getStatus()
    response.status(status).json({
      success: false,
      time: new Date(),
      data: exception.message,
      status,
      path: request.url
    })
  }
}

@Catch() 装饰器绑定所需的元数据到异常过滤器上。它告诉 Nest 这个特定的过滤器正在寻找 HttpException 而不是其他的。在实践中，@Catch() 可以传递多个参数，所以你可以通过逗号分隔来为多个类型的异常设置过滤器。

ts
//main.ts全局注册
app.useGlobalFilters(new HttpFilter())

管道 ​

管道是具有 @Injectable() 装饰器的类。管道应实现 PipeTransform 接口。

管道有两个典型的应用场景:

  • 转换：管道将输入数据转换为所需的数据输出 (例如，将字符串转换为整数)
  • 验证：对输入数据进行验证，如果验证成功继续传递; 验证失败则抛出异常

在这两种情况下, 管道 参数(arguments) 会由控制器 (controllers) 的路由处理程序行处理。Nest 会在调用这个方法之前插入一个管道，管道会先拦截方法的调用参数，进行转换或是验证处理，然后用转换好或是验证好的参数调用原方法。

Nest 自带很多开箱即用的内置管道。你还可以构建自定义管道。

内置管道 ​

Nest 自带九个开箱即用的管道，他们从 @nestjs/common 包中导出，即

管道名称功能说明
ValidationPipe对输入数据进行验证，确保其符合预定义的规则和结构，通常与类验证器一起使用
ParseIntPipe将输入的值转换为整数类型，如果转换失败则抛出异常
ParseFloatPipe将输入的值转换为浮点数类型，如果转换失败则抛出异常
ParseBoolPipe将输入的值转换为布尔值类型，支持多种布尔值的字符串表示
ParseArrayPipe将输入的值转换为数组类型，支持多种数组格式的字符串表示
ParseUUIDPipe验证输入的值是否为有效的 UUID 格式，并将其转换为字符串表示
ParseEnumPipe验证输入的值是否为指定枚举类型中的一个有效值
DefaultValuePipe如果输入的值为 undefined 或 null，则将其替换为预定义的默认值

  • @IsNumber()：验证属性是否为数字.
  • @Min() 和 @Max()：验证数字属性的最小值和最大值.
  • @IsArray()：验证属性是否为数组.
  • @IsEnum()：验证属性是否为枚举类型中的一个值。

可以向 CreateLoginDto 类添加一些装饰器，在 create-login.dto.ts 文件中。

ts
// create-login.dto.ts
import { IsNotEmpty, IsString, IsNumber, Length } from 'class-validator';
export class CreateLoginDto {
  @IsNotEmpty()
  @IsString()
  @Length(5, 10, {
    message: '长度在5到10之间'
  })
  name: string;
  @IsNotEmpty()
  @IsNumber()
  age: number;
}

+ class-transformer

class-transformer 是一个用于 TypeScript 的轻量级库，主要用于实现 JavaScript 普通对象（plain objects）和类实例之间的转换。它通过装饰器模式，使得开发者能够定义如何将对象属性从一个形式映射到另一个形式，以及在转换过程中如何处理复杂的类型和嵌套的对象结构。

GitHub - typestack/class-transformer: Decorator-based transformation, serialization, and deserialization between objects and classes.

核心功能

  • 序列化和反序列化：
    - plainToInstance（或 plainToClass）：将普通对象转换为类实例。例如，将从 API 接收的 JSON 数据转换为类实例.
    - classToPlain：将类实例转换为普通对象，通常用于将类实例序列化为 JSON 数据发送到前端.
  • 装饰器支持：
    - @Transform：允许在转换过程中为每个字段指定一个转换函数，例如将字符串转换为大写.
    - @Expose 和 @Exclude：用于控制哪些属性在序列化时被包含或排除.
ts
import { ArgumentMetadata, HttpException, HttpStatus, Injectable, PipeTransform } from '@nestjs/common';

import { plainToInstance } from 'class-transformer';
import { validate } from 'class-validator';
@Injectable()
export class LoginPipe implements PipeTransform {
  async transform(value: any, metadata: ArgumentMetadata) {
    console.log(value, metadata)
    const DTO = plainToInstance(metadata.metatype, value);
    const errors = await validate(DTO)
    console.log(errors);
    if (errors.length > 0) {
      throw new HttpException(errors, HttpStatus.BAD_REQUEST)
    }
    return value;
  }
}

守卫 ​

守卫是一个使用 @Injectable() 装饰器的类。 守卫应该实现 CanActivate 接口。

守卫有一个单独的责任。它们根据运行时出现的某些条件（例如==权限，角色，访问控制列表==等）来确定给定的请求是否由路由处理程序处理。这通常称为授权。在传统的 Express 应用程序中，通常由中间件处理授权 (以及认证)。中间件是身份验证的良好选择，因为诸如 token 验证或添加属性到 request 对象上与特定路由 (及其元数据) 没有强关联。

但是中间件不知道调用 next() 函数后会执行哪个处理程序。另一方面，守卫可以访问 ExecutionContext 实例，因此确切地知道接下来要执行什么。它们的设计与异常过滤器、管道和拦截器非常相似，目的是让您在请求/响应周期的正确位置插入处理逻辑，并以声明的方式进行插入。这有助于保持代码的简洁和声明性。

==守卫在每个中间件之后执行，但在任何拦截器或管道之前执行。==

  • 定义守卫
ts
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';
import { Observable } from 'rxjs';

@Injectable()
export class RoleGuard implements CanActivate {
  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {

    console.log('经过了守卫')
    return true;
  }
}
  • 绑定守卫
ts
import { RoleGuard } from './role/role.guard'
@Controller('guard')
@UseGuards(RolesGuard)
export class GuardController {}
  • 全局守卫
ts
//main.ts
const app = await NestFactory.create(AppModule);
app.useGlobalGuards(new RolesGuard());
  • 针对角色控制守卫
    Nest 提供了通过 @SetMetadata() 装饰器将定制元数据附加到路由处理程序的能力。这些元数据提供了我们所缺少的角色数据，而守卫需要这些数据来做出决策。
ts
import { Reflector } from '@nestjs/core';
//...
@Post()
@SetMetadata('roles', ['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}

设置守卫条件

ts
const roles = this.reflector.get<string[]>('roles', context.getHandler())
const req = context.switchToHttp().getRequest<Request>()
console.log('经过了守卫', roles, req.query)
if (roles.includes(req.query.role as string)) {
	return true;
} else {
	return false
}

自定义装饰器 ​

通过 createParamDecorator 定义

ts
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const User = createParamDecorator((data: string, ctx: ExecutionContext) => {
  const request = ctx.switchToHttp().getRequest();
  const user = request.user;

  return data ? user && user[data] : user;
});
  • 装饰器聚合
    Nest 提供了一种辅助方法来聚合多个装饰器。
ts
import { applyDecorators } from '@nestjs/common';

export function Auth(...roles: Role[]) {
  return applyDecorators(
    SetMetadata('roles', roles),
    UseGuards(AuthGuard, RolesGuard),
    ApiBearerAuth(),
    ApiUnauthorizedResponse({ description: 'Unauthorized"' })
  );
}

这具有通过一个声明应用所有四个装饰器的效果。

装饰器适用级别描述示例
@ApiOperation()Method用于描述 API 操作的详细信息，例如操作的名称、描述、标签等。@ApiOperation({ summary: 'Get user by ID' })
