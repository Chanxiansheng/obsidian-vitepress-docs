基于 Obsidian 多 Vault+ 软链接 +GitHubPages 的 VitePress 部署方案 ​

方案概览 ​

目录结构示例 ​

ts
├── main-vault/                  # 个人主 Obsidian Vault (独立Git仓库)
│   ├── 被链接目录1/
│   ├── 被链接目录2/
│   │   └── ...
│   └── .git/
│
├── vitepress-vault/             # VitePress 站点项目 (另一个Git仓库)
│   ├── docs/
│   │   ├── .vitepress/        
│   │   │   ├──config.mts
│   │   │   └── .dist/           # "docs:build" 输出目录
│   │   │
│   │   ├── index.md ->  
│   │   ├── symlink1 ->          # 软链接指向主 Vault 对应 被链接目录1
│   │   ├── symlink2 ->          # 软链接指向主 Vault 对应 被链接目录2
│   │   └── ... 
│   ├── package.json
│   ├── node_modules
│   ├── .git/
│   │
│   └── gh-pages/           # gh-pages 分支对应目录（通过 git worktree挂载）
│       └──...   ->         # 发布的静态文件手动或脚本复制到这里

核心设计点 ​

  1. 多仓库解耦：内容与发布职责分离

    • 主笔记仓库（main-vault）
      作为 Obsidian 主 Vault，独立 Git 仓库，负责内容创作和笔记管理。便于持续积累，不受站点结构干扰。
    • 站点构建仓库（vitepress-vault）
      专注于 VitePress 配置、构建、部署，独立 Git 仓库，避免笔记变动干扰站点生成逻辑。

  2. 软链接聚合内容：灵活、低耦合

    • 在 vitepress-vault/docs/ 中创建软链接指向 main-vault 中的特定子目录或内容。
    • 通过软链接聚合多个 Vault/子目录，可跨项目共用文档资源，实现模块化笔记组织。

  3. 自动侧边栏与索引：轻运维、结构清晰

    • 使用 vitepress-sidebar 插件自动生成 themeConfig.sidebar 配置。
    • 配合 Obsidian 插件（如 Templater）自动生成各目录 index.md 索引页面，便于导航。
    • 避免人工维护 Sidebar 配置，降低维护成本。

  4. Git Worktree 分支挂载：纯净部署，无需频繁切换

    • 使用 git worktree 技术将 gh-pages 分支挂载为独立目录（vitepress-vault/gh-pages）。
    • 构建产物直接复制至此目录后提交，即可发布站点。
    • 实现主分支开发和发布分支并行操作，避免手动切换、临时备份。

  5. 构建与发布逻辑明确

    • 本地运行 docs:build 输出至 .vitepress/.dist。
    • 发布时，清空 gh-pages/ 目录并复制构建产物，手动或脚本自动提交推送。
    • 保持构建产物目录干净，无软链接、无无关文件。

**软链接命令标准格式与参数**

Windows（命令提示符 cmd）

mklink [选项] <链接> <目标>
  • /D ：创建目录（文件夹）符号链接
  • /H ：创建硬链接（仅限文件）
  • /J ：创建目录联接点（junction）

Windows（PowerShell）

sh
New-Item -ItemType SymbolicLink -Path ".\C-工具说明书" -Target "A:\!Ob个人仓库\C-工具说明书"

Linux / macOS（终端）

ln -s <目标> <链接>
  • -s ：创建符号链接（软链接）

备注

  • Windows 需管理员权限或具备创建符号链接权限。
  • 路径可为绝对或相对路径。
  • 可链接文件或目录（Windows 目录需加 /D）。

遇到的问题：软链接目录未被识别

在 VitePress 中尝试使用软链接（symlink）指向公共内容目录（如 docs/shared → ../common-docs），发现插件不识别软链接内容，导致该目录未出现在侧边栏中。

通过源码发现，globSync 函数没用开启 follow，选项。

ts
globSync('**', {
  cwd: currentDir,
  maxDepth: 1,
  ignore: options.excludePattern || [],
  dot: true,
  follow: true // patch
});

目前，已提交了 Issue。

4. 文件夹路由 ​

  • 目标： 在访问 /folder/ 路径时自动呈现 folder/index.md 页面。

  • 实现方式：

    • 在 Obsidian 中使用 Templater 插件或自动化工具，为每个需要目录路由的文件夹生成一个 index.md 文件。
    • index.md 充当该目录的「路由入口」，在 VitePress 中就可以通过自动访问到。

  • 特点：深层递归、自动编码、自动名称转换

js
	<%*
	const encodePath = (p) => encodeURI(p);
	
	const currentDir = tp.file.folder(true);
	
	function trimSlash(path) {
	  return path.endsWith('/') ? path.slice(0, -1) : path;
	}
	
	function getRelativePath(from, to) {
	  from = from || '.';
	  to = to || '.';
	
	  if (from !== '.' && from.endsWith('/')) from = from.slice(0, -1);
	  if (to !== '.' && to.endsWith('/')) to = to.slice(0, -1);
	
	  const fromParts = from === '.' ? [] : from.split('/');
	  const toParts = to === '.' ? [] : to.split('/');
	
	  while (fromParts.length && toParts.length && fromParts[0] === toParts[0]) {
	    fromParts.shift();
	    toParts.shift();
	  }
	
	  const ups = Math.max(0, fromParts.length - 1);
	  const relPath = '../'.repeat(ups) + toParts.join('/');
	
	  return relPath || './';
	}
	
	// 返回值改为包含生成内容和是否有内容的标记
	async function generateIndex(basePath, relativeToPath, level = 2) {
	  let output = '';
	  const res = await app.vault.adapter.list(basePath);
	
	  const mdFiles = res.files.filter(f =>
	    f.endsWith('.md') &&
	    !f.endsWith('index.md') &&
	    !f.includes('_Index_of_') &&
	    !f.includes('.template.') &&
	    !f.includes('.excalidraw')
	  );
	
	  // 生成当前目录的文件列表
	  for (const file of mdFiles.sort()) {
	    const fileName = file.split('/').pop().replace(/\.md$/, '');
	    const relativePath = getRelativePath(relativeToPath, file);
	    output += `- [${fileName}](${encodePath(relativePath)})\n`;
	  }
	
	  // 递归生成子文件夹内容
	  for (const folder of res.folders.sort()) {
	    const { content: childContent, hasContent } = await generateIndex(folder, relativeToPath, level + 1);
	    if (hasContent) {
	      const folderName = folder.split('/').pop();
	      const headingLevel = Math.min(level, 6);
	      output += `\n${'#'.repeat(headingLevel)} ${folderName}\n\n`;
	      output += childContent;
	    }
	  }
	
	  // 是否有内容判断：当前目录有文件，或者子目录有内容
	  const hasContent = output.trim().length > 0;
	
	  return { content: output, hasContent };
	}
	
	let content = `# ${tp.file.folder(false)}\n\n> 本目录内容索引：\n\n`;
	const result = await generateIndex(currentDir, currentDir, 2);
	content += result.content;
	tR = content;
	%>

使用及发布流程建议 ​

1. 初始化 gh-pages 分支的 git worktree ​

假设你在 vitepress-vault 根目录：

bash
git worktree add gh-pages origin/gh-pages

如果远程没有 gh-pages，先创建空分支：

bash
	git checkout --orphan gh-pages
	git rm -rf .
	git commit --allow-empty -m "Init gh-pages branch"
	git push origin gh-pages
	git checkout main
	git worktree add gh-pages origin/gh-pages

这样 gh-pages 分支的工作区就在 vitepress-vault/gh-pages/ 目录。

2. 本地开发和构建 ​

  • 在 vitepress-vault 根目录进行日常开发，软链接保持和 main-vault 内容同步。

  • 执行构建：

    pnpm docs:build

  • 构建输出文件放在

    swift
    `vitepress-vault/docs/.vitepress/.dist/`

3. 发布步骤（手动或脚本自动化） ​

bash
# 进入 gh-pages 目录
cd vitepress-vault/gh-pages

# 清空旧内容（除 .git 目录）
git rm -rf .

# 复制构建产物（保持当前目录为 gh-pages 根目录）
cp -r ../docs/.vitepress/.dist/* .

# 添加变更，提交并推送
git add .
git commit -m "Deploy static site at $(date)"
git push origin gh-pages

# 返回主分支工作区
cd ..
git checkout main

Windows PowerShell 下 cp 和 rm 命令可以用对应的 Copy-Item 和 Remove-Item，或在 Git Bash 下使用 Linux 命令。

总结 ​

  • 主分支和 gh-pages 工作区完全分开，避免互相干扰。
  • 软链接保持源码同步，构建产物纯净，无软链接问题。
  • 手动控制发布时机，操作灵活，方便调试。
  • 使用 git worktree，无需频繁切换分支，工作区多且互不干扰。
